# PRD: HRD-Net 훈련 과정 성과 분석 대시보드

## 1. 문제 정의

### 사업 수익 구조

| 항목 | 수치 |
|---|---|
| 1개 기수 정원 | 30명 |
| 1인당 6개월 훈련비 | 17,424,000원 |
| 1인 1일 출석 시 훈련비 | 145,200원 |
| **기수당 최대 매출** | **약 5.2억원** |
| 누적 운영 규모 | 25기 / 2023.07~ |
| 사업 성과 | 전국 KDT 상위 2.3\~3.9% (2023\~2025) |

> 수익 = 모집 인원 × 출석률 × 수료율.
> 결석 1건 = 145,200원 매출 손실. 1명 중도탈락 = 최대 1,742만원 손실.
> **최대한 많은 수강생을 모집하고, 최대 출석률·수료율을 달성하는 것이 핵심 성과.**

### 배경

**기존에 어떻게 하고 있었는가?**
- HRD-Net 웹사이트에서 수동으로 데이터를 확인하고, **엑셀 수작업으로 정리하여 보고**
- 이후 HRD-Net 훈련기관 API를 발견하여 훈련 데이터를 엑셀로 저장하는 방식으로 개선 시도
- 그러나 매번 API를 수동으로 호출해서 내려받아야 했고, 데이터 간 연결성이 부족

**어떤 문제가 있었는가?**
- **체계적 모니터링 부재**: 출결·수료·시장 데이터가 각각 흩어져 있어 통합된 현황 파악 불가
- **보고 지연**: 엑셀 수작업 → 보고서 작성까지 시간 소요, 실시간 대응 불가능
- **위험군 조기 감지 불가**: 결석 누적, 이탈 징후를 사후에야 파악 → 개입 시점을 놓침
- **시장 인사이트 부재**: 경쟁 과정 현황, 시장 트렌드를 파악할 수단 없음 → 다음 기수 기획이 감(感)에 의존
- **정보 비대칭**: 데이터를 담당자 1명만 볼 수 있어 구성원 전체의 상황 인식 불일치

**그 문제로 인한 비용/손실은?**
- **시간**: 주간/월간 보고 자료 작성에 수시간 소요 (데이터 수집 → 엑셀 정리 → 보고서 포맷팅)
- **매출**: 위험군 조기 감지 실패 → 중도탈락 1명당 최대 1,742만원 손실
- **의사결정 지연**: 실시간 현황이 없어 출결 이상 감지~조치까지 수일 지연
- **기회비용**: 시장 동향 분석 없이 다음 기수 과정 기획 → 모집률·경쟁력 저하 위험

## 2. 목표

### 정성 목표
- 운영 담당자·강사·경영진 **전원이 동일한 데이터**를 보고 같은 맥락에서 의사결정할 수 있는 환경 구축
- 출결 이상·위험군을 **즉시 감지**하여 중도탈락을 사전에 방지하는 체계 마련
- 시장 동향 데이터를 기반으로 **다음 기수 기획에 근거 있는 의사결정**이 가능한 상태

### 정량 목표
| 지표 | Before | After (목표) | After (실제) |
|---|---|---|---|
| 보고 소요 시간 | 반나절(4h+) 엑셀 수작업 | 실시간 (대시보드 접속 즉시) | **0분** — 대시보드 접속 즉시 확인, 보고용 텍스트 자동 생성 |
| 위험군 감지 시점 | 사후 파악 (수일 지연) | 당일 자동 감지 | **당일 자동 감지** — 결석 3회+, 지각 5회+, 조퇴 5회+ 자동 플래그 |
| 데이터 수집 | 수동 API 호출 → 엑셀 저장 | 자동 ETL | **평일 매시간 자동 수집** — GitHub Actions 스케줄 운영 중 |
| 시장 분석 데이터 | 0건 | 30만건+ | **30만건+ 축적** — 매일 21시 자동 수집 지속 |
| 정보 접근성 | 담당자 1명만 보유 | 구성원 전원 접근 | **전원 접근** — URL 공유로 운영·강사·경영진 동시 열람 |

## 3. 사용자

### 사용자 유형
| 사용자 | 역할 | 주요 관심사 | 사용 빈도 |
|---|---|---|---|
| 운영 담당자 | 훈련 과정 운영·관리, 보고 | 오늘의 출결 현황, 위험군 감지, 수료율 관리 | 매일 |
| 강사 | 수업 진행, 학생 관리 | 학생별 출결 현황, 결석·지각 패턴, 체류시간 | 매일~주 수회 |
| 경영진 | 전체 성과 검토, 다음 기수 의사결정 | KPI 요약(수료율·취업률), 시장 동향, 경쟁 과정 비교 | 주 1회~수시 |

### 사용자 페인포인트
1. **운영 담당자**: 엑셀 수작업 보고에 반나절 소요, 실시간 현황 파악 불가 → 이상 징후 대응 지연
2. **강사**: 개별 학생의 출결 누적 현황을 한눈에 볼 수 없어, 면담·개입 타이밍을 놓침
3. **경영진**: 경쟁 과정 대비 우리 과정의 위치(모집률, 수료율, 취업률)를 파악할 데이터 부재 → 다음 기수 기획이 감에 의존

## 4. 핵심 기능 및 우선순위

| 우선순위 | 기능 | 선정 근거 |
|---|---|---|
| P0 (필수) | ETL 자동화 (출결·과정·훈련생 데이터 자동 수집) | 수작업 제거가 전제 조건. 데이터 없으면 대시보드 자체가 불가능 |
| P0 (필수) | 오늘의 출결 현황 대시보드 (home) | **매일 가장 먼저 보는 화면**. 출석률·결석·입실중 실시간 파악 → 즉시 대응 |
| P0 (필수) | 누적 출결 위험 지표 (결석 3회+, 지각 5회+, 조퇴 5회+) | 중도탈락 예방이 직접 매출과 연결 (1명 탈락 = 최대 1,742만원 손실) |
| P1 (중요) | 기수별 분석 (인구통계, 요일별 패턴, 학생별 현황) | 기수 운영 중 개선 포인트 발견, 강사·운영 담당자의 학생 관리 지원 |
| P1 (중요) | 진행과정 관리 (출석률 게이지, 보고용 텍스트, 추이 차트) | 경영진 보고를 대시보드로 대체 → 보고 소요 시간 반나절 → 0 |
| P2 (있으면 좋음) | 시장 동향 분석 13탭 (경쟁 과정, 비용-성과, 자격증 분석, 사업기회 발굴 등) | 다음 기수 기획 시 데이터 기반 의사결정 지원. 30만건+ 외부 데이터 활용 |
| P2 (있으면 좋음) | 데이터 감사 (원본 데이터 조회, 수동 ETL) | 데이터 정합성 검증, 운영 담당자의 디버깅·확인 용도 |
| P2 (있으면 좋음) | SQL Playground (SELECT 전용 쿼리 실행) | 실제 데이터로 SQL 학습·탐색, 예제 쿼리(초급/중급/고급) 제공 |
| P2 (있으면 좋음) | SQL Playground (SELECT 전용 쿼리 실행) | 실제 데이터로 SQL 학습·탐색, 예제 쿼리(초급/중급/고급) 제공 |

### 우선순위 판단 기준
- **P0**: 매출에 직접 영향 (출석 = 훈련비, 탈락 = 매출 손실) + 매일 사용하는 핵심 기능
- **P1**: 운영 효율화 (보고 시간 절감) + 기수 운영 중 의사결정 지원
- **P2**: 중장기 전략적 가치 (시장 분석, 데이터 품질 관리) — 당장 매출과 직결되지 않지만 경쟁력 강화에 기여

## 5. 성공 지표

### 북극성 지표
**지표:** 기수별 수료율
**이유:** 수료율은 모집 → 출석 관리 → 이탈 방지의 전 과정이 반영된 최종 결과 지표. 수료율이 높다 = 모집 잘 하고, 출석 관리하고, 중도탈락을 막았다는 의미. 직접적으로 매출(수료 인원 × 훈련비)과 연결.

### KPI
| KPI | 측정 방법 | 목표치 | 현재 |
|---|---|---|---|
| 일일 출석률 | 홈 > 오늘의 출결 현황 (출석 / 전체) | 90% 이상 유지 | 대시보드에서 실시간 확인 가능 |
| 누적 위험군 비율 | 진행과정 관리 > 위험 지표 (결석 3회+ / 전체) | 10% 이하 | 대시보드에서 실시간 확인 가능 |
| 기수별 수료율 | 홈 > KPI 요약 (수료인원 / 수강인원) | 80% 이상 | 대시보드에서 기수별 확인 가능 |
| 보고 소요 시간 | 보고 자료 준비에 걸리는 시간 | 0분 (대시보드 대체) | **0분 달성** |

### Metric (참고 지표)
| Metric | 용도 |
|---|---|
| 취업률 (3개월/6개월 고용보험) | 과정 품질의 후행 지표. 직접 통제 어렵지만 성과 비교의 기준 |
| 시장 내 우리 과정 순위 (모집률, 취업률) | 경쟁 포지션 파악. 다음 기수 기획 시 참고 |
| 지각·조퇴 건수 추이 | 출석률만으로 안 보이는 패턴 감지 (지각 증가 = 이탈 전조) |
| 체류시간 (입실~퇴실) | 형식적 출석 vs 실질적 학습 참여 구분 |

## 6. 제약 조건

### 기술적 제약
- **Streamlit 단방향 구조**: 읽기 전용 대시보드만 가능, 데이터 입력/수정 기능 구현에 한계
- **무료 인프라 의존**: Streamlit Cloud(무료) + Supabase(무료 티어) → 트래픽·용량 제한 존재
- **ETL 실행 환경**: GitHub Actions 무료 시간 한도 내 운영 (평일 매시간 + 매일 1회)

### 조직적 제약
- **1인 개발**: 기획·개발·운영을 혼자 수행 → 기능 우선순위를 엄격히 관리해야 함
- **훈련 운영과 병행**: 개발 전담이 아닌 훈련 운영 업무와 병행하며 진행

### 데이터 제약
- **HRD-Net API 의존**: 공공데이터 API의 응답 형식·가용성에 종속. API 장애 시 데이터 수집 중단
- **취업률 후행성**: 고용보험 기반 취업률은 수료 후 3~6개월 뒤 확정 → 실시간 성과 측정 불가
- **개인정보 제한**: 훈련생 식별 정보는 코드(CSTMR_ID)로만 제공, 상세 개인정보 접근 불가

## 7. 비즈니스 임팩트 (회고)

### 의사결정 사례

**사례 1: 출결 위험군 조기 감지 → 즉시 상담 → 맞춤 대응**
- 상황: 이탈·결석·지각·조퇴는 즉시 매출 타격이므로 **긴급 1순위** 대응 과제
- 발견: 누적 출결 위험 지표(결석 3회+, 지각 5회+)에서 위험군 자동 감지
- 조치: 감지 즉시 1:1 상담을 진행하고, 원인별 맞춤 대응 실시
  - **학습 부진** → 보충 강의 제공, 학습 방향 재설계
  - **진로 고민** → 시장 분석 데이터 기반으로 취업 방향 조언
  - **자격증 어려움** → 직접 자격증 취득(정보처리기사, 리눅스마스터, ADsP 모두 취득) 후 공부법·후기 안내
  - **통학 어려움** → 부동산 시세, 고시원 등 주거 대안까지 직접 탐색하여 제공
- 결과: 기존에는 이탈 징후를 사후에야 파악해 속수무책이었으나, **조기 감지 → 즉시 대응 체계** 구축으로 이탈 의사가 있는 훈련생에게 다양한 액션을 취할 수 있게 됨. 대응에도 불구하고 이탈한 케이스도 있지만, **위험군 감지 → 대응 → 기록의 사이클** 자체가 운영 역량으로 축적됨

**사례 2: 시장 동향 분석 → 경쟁사 벤치마킹 → 복합적 운영 개선**
- 상황: 다음 기수 기획 시 모집·마케팅·커리큘럼·운영 전반의 전략이 필요했음
- 발견: 30만건+ 시장 데이터에서 모집 잘 되는 커리큘럼 트렌드, 경쟁사 우수 사례, 독보적 과정 등을 분석
- 조치 및 결과:
  - **커리큘럼 트렌드 대응**: AI·파이썬이 시장 대세임을 데이터로 확인 → 본인이 직접 파이썬 학습 후 이 프로젝트를 1인 기획~배포까지 완수하며 다음 스텝 대비
  - **교육 플랫폼 개선**: 경쟁사 분석 결과 반영 → 레거시 강의 → 인프런 최신 강의로 교체, 비선호 코딩테스트 플랫폼(구름) → 백준·프로그래머스(PCCE, PCCP, PCSQL)로 전환
  - **마케팅 차별화**: 100% 온라인 과정이 장점을 홍보하는 것을 발견 → 우리 과정의 **100% 오프라인 강점**을 반대 포지셔닝으로 마케팅
  - **개강 시기 전략**: 삼성 SSAFY 등 대형 과정의 개강 시기를 분석하여 경쟁이 약한 시기에 개강하는 인사이트 도출
  - **운영 벤치마킹**: 웰컴키트·굿즈(옷, 스티커) 제작, 크리스마스 이벤트·조기 출석 이벤트 등 타 오프라인 과정의 우수 운영 사례를 발견하고 도입
  - **장기 과정 설계 인사이트**: SAP처럼 독보적 기술로 100% 모집을 달성하는 과정을 발견 → 추후 과정 설계 시 차별화된 기술 스택 도입 검토
